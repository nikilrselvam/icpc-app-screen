I don't believe computer scientists strictly learn CS to learn to code. Computer scientists are inherently problem solvers, and the code we write are the solutions. Learning about data structures and algorithms that assist in solving these problems is something I have been struck with a desire for. I want to gain to knowledge in order to not only apply these problems in future programs and potentially in competitions, but also to share this knowledge with others and hopefully inspire others to strive to be better problem solvers and find excitement in building excellent code with new-found algorithmic thinking. The ICPC workshops have sparked a huge interest for this field in me and I hope to do the same to others while in the process improving my algorithmic thinking as well.

#### What’s your favorite problem? It does not have to be an algorithm, or even CS related!
>While I am still exploring all the algorithm problems that CS has to offer, the one that I first heard about that intrigued me is the open P vs. NP problem. I'm not sure if this counts as a problem, per say, since it is still an unsolved problem, but it's extremely intriguing. In essence, the question is whether a problem whose solution is quickly verifiable can also be solved quickly. While P problems can be solved and verified quickly, NP problems can be verified quickly, but the problems have an unknown difficulty. To be in set P means that it runs in polynomial time. NP, which stands for nondeterministic polynomial time, is not immediately assumed to be more "difficult". Rather, there is no known polynomial time solution to these problems, if it exists at all. The problem is whether P = NP or P != NP. While problems in P must be in NP, the issue is whether problems in NP can lie in P. This problem has existed for nearly 50 years, and a possible solution to this would have massive effects on multiple fields, including cryptography, game theory, AI, mathematics, and many more.
